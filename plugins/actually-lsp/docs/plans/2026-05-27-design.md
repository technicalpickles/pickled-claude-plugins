# actually-lsp design spec

**Date:** 2026-05-27
**Status:** draft, awaiting review
**Companion docs:**
- [CONTEXT.md](../../CONTEXT.md): vocabulary, state machine, relationships
- [ADR 0001](../adr/0001-layer-on-top-of-official-lsp-plugins.md): layered on top of official LSP plugins

## Goal

Close the LSP activation gap across the whole lifecycle (detection, setup, activation, failure diagnosis) without requiring users to write per-prompt preambles by hand. Operationalizes the research in [`projects/claude-test-harness/docs/2026-05-21-lsp-activation-gap-share-out.md`](../../../../../projects/claude-test-harness/docs/2026-05-21-lsp-activation-gap-share-out.md) and the setup steps in [`2026-05-22-lsp-in-claude-code-howto.md`](../../../../../projects/claude-test-harness/docs/2026-05-22-lsp-in-claude-code-howto.md).

## v1 scope

Three ecosystems supported on day one: Rust, TypeScript, Ruby. One recommended LSP plugin per ecosystem (from `claude-plugins-official`), with detection structured to allow alternates later. See [CONTEXT.md](../../CONTEXT.md) for the canonical state machine and terms.

### In v1

- SessionStart hook: per-ecosystem detection, nudge or activation context based on LSP state
- PreToolUse hook: deferred detection for paths Claude touches outside the initial cwd's detected scope
- PostToolUse hook: failure context on LSP tool errors, with cache invalidation
- `/actually-lsp-doctor` slash command: diagnose + fix, narrow with `fix` and `<ecosystem>` args
- `/actually-lsp-ignore` slash command: dismissal
- Project state file at `<project>/.claude/actually-lsp.json`
- Session state file at `~/.cache/actually-lsp/sessions/<session-id>.json`
- Trust-but-verify cache invalidation: marker mtime, `claude plugin list --json` hash, PostToolUse failure signal

### Deferred (post-v1)

- `permission-missing` as a distinct state (folded into `server-not-runnable` for v1)
- Auto-recovery in the PostToolUse hook
- Time-based cache TTL (signals-only invalidation for v1)
- Per-language sibling plugins (`actually-rust-lsp`, etc.). Rust is the likely first split if `initializationOptions` warm-up config becomes a recurring user need.
- `/actually-lsp-unignore` command to reverse dismissals
- Additional ecosystems (Go, Python, etc.) beyond the three official ones

## Hook scripts

### `session-start.sh`

Fires once per Claude Code session at boot. Runs the full detection pass for the initial cwd, writes project state, populates session state, emits per-ecosystem output as `additionalContext`.

Steps:
1. Read `$CLAUDE_PROJECT_DIR` (or fall back to `$PWD`).
2. Source `lib/detect.sh`, run `detect_ecosystems "$project_dir"`. Returns a list of `(ecosystem, marker_path)` pairs.
3. Source `lib/state.sh`, load existing project state file if present.
4. Source `lib/ecosystems.sh` for per-ecosystem checks.
5. For each detected ecosystem, run trust-but-verify:
   - Compare marker mtime to cached `last_marker_mtime`
   - Run `timeout 5s claude plugin list --json`. On success, compare hash to cached `last_plugin_list_hash`. On timeout or non-zero exit, skip the hash check, trust the cached `last_plugin_list_hash`, and append `(unverified: plugin list query failed)` to any emitted output for transparency.
   - If marker mtime or plugin list hash changed, recompute state
   - Otherwise, trust cached state
6. Emit per-ecosystem output:
   - `dismissed`: silent
   - `not-detected`: skipped (filtered in step 2)
   - `no-lsp-plugin`: nudge ("Detected Rust. Recommended LSP plugin: rust-analyzer-lsp@claude-plugins-official. Run `/actually-lsp-doctor` to set up.")
   - `server-not-runnable`: nudge ("rust-analyzer-lsp installed but Rust env not ready. Run `/actually-lsp-doctor` to fix.")
   - `ready`: read `activation/<ecosystem>.md`, emit verbatim as activation context
   - `error`: surface the error, point at `/actually-lsp-doctor <ecosystem>`
7. Write fresh project state file with `last_check_at`, `last_marker_mtime`, `last_plugin_list_hash`, `state`, `dismissed` per ecosystem.
8. Write session state file under `~/.cache/actually-lsp/sessions/$CLAUDE_SESSION_ID.json` with `nudged_ecosystems: ["rust", "ruby"]` (so PreToolUse can dedup).

Polyglot output shape: group by ecosystem, summary line first, activation context for `ready` ecosystems appended after. Example:

```
[actually-lsp]
Rust: ready
Ruby: no LSP plugin installed. Recommend `/plugin install ruby-lsp@claude-plugins-official`

[Rust activation context]
...
```

### `pre-tool-use.sh`

Fires on every PreToolUse event. Job: deferred detection for paths Claude touches that aren't covered by SessionStart's initial scan.

Steps:
1. Read session state file. If unreadable (session hasn't seen SessionStart yet), exit silently. SessionStart will catch up.
2. Parse the tool call's args to extract candidate paths:
   - `Read`, `Edit`, `Write`: `file_path`
   - `Bash`: parse for `cd <target>`; fall back to the first absolute path in the command
   - `Glob`, `Grep`: `path` arg
3. For each candidate path, walk up looking for ecosystem markers (closest ancestor `Cargo.toml` / `package.json` / `Gemfile` defines the project root for that ecosystem).
4. If a (root, ecosystem) pair is found AND it's not in the session state's `nudged_ecosystems` list:
   - Run state detection (full path, including trust-but-verify against the project state file at that root)
   - Emit `additionalContext` for the state (same content shape as SessionStart, scoped to this one ecosystem)
   - Add the pair to session state's `nudged_ecosystems`
5. For `ready` state at this hook position: include the activation context, plus a one-line caveat: "Activation context at PreToolUse position lands soft (0.4/5 per share-out). For inline-rate activation (4.6/5), run `/actually-lsp-doctor` to load the context at user-prompt position."

**Performance**: the filesystem walk in step 3 happens BEFORE the dedup check, so it runs on every PreToolUse fire even after the (root, ecosystem) is already in `nudged_ecosystems`. v1 ships without optimization. PR 3 includes a benchmark that measures cumulative hook overhead across a representative Claude Code session. If the benchmark shows >500ms cumulative or >50ms p99 per fire, add per-directory memoization to session state: once a directory is known to belong to a given (root, ecosystem) pair, paths under it skip the walk.

**Dedup robustness**: session state writes must be atomic (write to temp file, rename). On parse failure when reading, treat as "no dedup info available, fire once and rewrite from scratch". The single bonus fire is acceptable; runaway re-fires aren't.

### `post-tool-use.sh`

Fires on every PostToolUse event. Job: catch LSP tool failures, emit failure context, invalidate cache.

Steps:
1. Read tool name from event. If not `LSP`, exit.
2. Read tool result. If not an error, exit.
3. Identify the ecosystem from the LSP call's `lspServer` arg (e.g., `rust-analyzer` -> Rust). Source `lib/ecosystems.sh` for the mapping.
4. Emit `additionalContext`:
   ```
   [actually-lsp] LSP call to <method> on <server> failed:
   <error message verbatim>

   Cache for <ecosystem> has been invalidated. Run `/actually-lsp-doctor <ecosystem>` to re-verify.
   ```
5. Update project state file: set `<ecosystem>.state = "error"`, set `<ecosystem>.last_error = <error>`. Don't run full detection here. Keep the hook cheap. The next SessionStart or PreToolUse will re-verify.

## Skills (user-invocable slash actions)

These are skills, not plugin commands. Claude Code surfaces skills for `/<skill-name>` invocation; plugin `commands/<name>.md` files are not surfaced and are not used here.

Skill names are globally namespaced (the system surfaces them by their bare `name:` slug, not `plugin:name`), so the skills here are prefixed with `actually-lsp-` to avoid colliding with other plugins' `doctor` or `ignore` skills.

### `skills/actually-lsp-doctor/SKILL.md`

Skill invoked by `/actually-lsp-doctor [fix] [<ecosystem>]`.

The SKILL.md body is markdown that becomes Claude's instructions when the skill activates. The frontmatter `description:` should be a "use when X" sentence so Claude reaches for the skill in the right contexts (post-nudge, missing plugin, env not ready, explicit invocation). Outline:

```markdown
You are running /actually-lsp-doctor. Parse the user's args:
- "fix" as the first arg means skip the diagnostic report, jump to action
- "rust" | "typescript" | "ruby" as an arg narrows to that ecosystem
- no args means full diagnostic + interactive fix

Step 1: Read the project state file at .claude/actually-lsp.json. If missing, run detection now (see lib/detect.sh).

Step 2 (diagnose, unless `fix` arg): output a per-ecosystem report:
- State per ecosystem
- For non-ready ecosystems, what's needed
- For dismissed ecosystems, note them

Step 3 (act): for each ecosystem in `no-lsp-plugin`, `server-not-runnable`, or `error`:
- `no-lsp-plugin`: try `claude plugin install <recommended>@claude-plugins-official` via Bash. The user gets a permission prompt. If denied, output the slash command `/plugin install <recommended>@claude-plugins-official` and ask the user to run it themselves.
- `server-not-runnable`: run env fixes per ecosystem via Bash. Bundle install / cargo build / npm install auto-run (user has implicit project consent). Global gem installs auto-run too. Per project-level decision, all env fixes auto-run.
- `error`: surface the cached error, ask the user how to proceed.

Step 4: re-run detection, update project state file, report final state.

Tone: terse, no celebration messaging. Just status and next step.
```

### `skills/actually-lsp-ignore/SKILL.md`

Skill invoked by `/actually-lsp-ignore [<ecosystem>]`.

```markdown
You are running /actually-lsp-ignore. Parse args:
- "rust" | "typescript" | "ruby" as an arg dismisses that one directly
- no args: read project state, present detected ecosystems, ask the user which to dismiss

Update the project state file at .claude/actually-lsp.json: set `<ecosystem>.dismissed = true`.

Output a one-line confirmation: "Ignoring <ecosystem> for this project. To re-enable later, manually edit .claude/actually-lsp.json (a re-enable command is a future extension)."
```

## State file schemas

### Project state file: `<project>/.claude/actually-lsp.json`

```json
{
  "version": 1,
  "ecosystems": {
    "rust": {
      "state": "ready",
      "dismissed": false,
      "last_check_at": "2026-05-27T14:00:00Z",
      "last_marker_mtime": "2026-04-15T10:22:00Z",
      "last_plugin_list_hash": "abc123",
      "last_error": null
    },
    "ruby": {
      "state": "no-lsp-plugin",
      "dismissed": false,
      "last_check_at": "2026-05-27T14:00:00Z",
      "last_marker_mtime": "2026-03-01T08:00:00Z",
      "last_plugin_list_hash": "abc123",
      "last_error": null
    }
  }
}
```

Gitignored by default (the plugin's install instructions will recommend adding it to project `.gitignore`). A team that wants shared dismissal state can commit it.

**Location rationale**: `<project>/.claude/actually-lsp.json` lives inside Claude Code's existing convention area but uses a plugin-namespaced filename, which doesn't collide with Claude Code's path-segment-based conventions (`.claude/settings.json`, `.claude/commands/`, `.claude/skills/`, `.claude/hooks/`, `.claude/plugins/`). In-repo location keeps the team-shareable property. If a future Claude Code convention conflicts, migrating to `<project>/.claude/plugins/actually-lsp/state.json` is a mechanical version-1 -> version-2 schema bump.

### Session state file: `~/.cache/actually-lsp/sessions/<session-id>.json`

```json
{
  "session_id": "f787146a-c98b-44e3-9496-1ced1da40da2",
  "nudged_ecosystems": [
    {"project_root": "/Users/josh.nichols/pickleton/repos/welcome2u/worktrees/main", "ecosystem": "rust"},
    {"project_root": "/Users/josh.nichols/pickleton/repos/bktide/worktrees/main", "ecosystem": "typescript"}
  ]
}
```

Garbage collection: on SessionStart, the hook deletes session state files older than 30 days. Self-cleaning, no separate cron needed.

## Detection logic

`lib/ecosystems.sh` defines per-ecosystem records:

```bash
# Format: ecosystem|marker|recommended_plugin|server_binary|env_check_cmd
ecosystems=(
  "rust|Cargo.toml|rust-analyzer-lsp@claude-plugins-official|rust-analyzer|cargo metadata --no-deps"
  "typescript|package.json|typescript-lsp@claude-plugins-official|typescript-language-server|test -d node_modules"
  "ruby|Gemfile|ruby-lsp@claude-plugins-official|ruby-lsp|bundle check"
)
```

The TypeScript ecosystem check should require both `package.json` AND at least one `.ts` or `.tsx` file in the tree (to avoid nudging on pure JavaScript projects).

State derivation per ecosystem:
1. Run marker check. If absent, state = `not-detected`.
2. Check dismissed flag in project state file. If true, state = `dismissed`.
3. Run `claude plugin list --json | jq '.[] | select(.enabled and .id == "<recommended_plugin>")'`. If empty, state = `no-lsp-plugin`.
4. Run `command -v <server_binary>`. If absent, state = `server-not-runnable`.
5. Run env_check_cmd. If non-zero exit, state = `server-not-runnable`.
6. Otherwise, state = `ready`.

## Activation context

Per-ecosystem markdown in `activation/<ecosystem>.md`. The content is the share-out's preamble, tailored per ecosystem (operations, server name, plugin name). Drafts to start from:
- bktide cell-2 preamble for TypeScript (4.6/5 at N=5)
- welcome2u cell-2 preamble for Rust (with the warm-up caveat appended per ADR-locked decision)
- Adapt the Ruby variant from the how-to's example

For Rust, the activation context includes a final paragraph: "If `cargo build` hasn't run recently, run it before LSP queries. rust-analyzer's `workspace/symbol` races indexing on cold cache."

**Size budget**: each activation context is ~250 tokens (the share-out's measured preamble). A polyglot session with three `ready` ecosystems contributes ~750 tokens at SessionStart. This is <1% of a typical session context window and not a v1 concern. PreToolUse dedup (see the pre-tool-use.sh "Dedup robustness" note) prevents repeated re-emission. Compression experiments are deferred: any shorter variant would need to replicate the share-out's methodology to confirm activation rate doesn't regress, and that's a separate research project.

## Testing strategy

Three layers:

1. **Unit-ish tests for detection**: shell test scripts in `tests/` that set up fake project directories with various marker file combinations, run detection, assert state output. Per-ecosystem coverage.
2. **Hook output tests**: scripts that simulate hook invocations (set env vars, run hook script, capture stdout), assert the expected `additionalContext` shape per state.
3. **Smoke test against real projects**: documented manual test plan that runs `actually-lsp` against bktide (TypeScript), welcome2u (Rust), homesick (Ruby), the same projects from the share-out probes, and confirms LSP usage rates remain in line with the share-out's data (TypeScript ~4.6/5, Rust ~3/5 with warm-up).

## First PR shape

Stop here when CI is green. Don't try to land everything in one PR.

PR 1: skeleton + detection + SessionStart hook for one ecosystem (TypeScript, since it has the strongest activation data). Just the happy path: detect, nudge or activate, write project state. No PreToolUse, no PostToolUse, no slash commands yet. Goal: prove the architecture in production on one project, get feedback before scaling.

PR 2: add Rust + Ruby detection + their activation context files.

PR 3: PreToolUse hook for deferred detection.

PR 4: PostToolUse hook for failure context.

PR 5: slash commands (`/actually-lsp-doctor`, `/actually-lsp-ignore`).

PR 6: tests + docs + README.

Each PR includes a version bump per the marketplace convention (`./scripts/bump-version.sh actually-lsp minor` or `patch`).

## Resolved during 2026-05-27 review

Four questions were open at first draft. Resolutions folded into the relevant sections above:

- **PreToolUse performance budget**: ship without memoization, benchmark in PR 3. See pre-tool-use.sh "Performance" note.
- **Activation context size**: ship full 250-token preamble per ecosystem with robust dedup. See Activation context "Size budget" note.
- **`claude plugin list --json` reliability**: 5-second timeout with cache fallback, "unverified" flag on degraded output. See session-start.sh step 5.
- **Marker file location**: keep `<project>/.claude/actually-lsp.json`; migration path documented. See State file schemas "Location rationale" note.
